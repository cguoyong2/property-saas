# 数据库初始化与迁移规范

## 1. 本地初始化方式

本地开发优先使用根目录 `docker-compose.yml` 启动 MySQL 8、Redis 7、MinIO：

```bash
docker compose up -d mysql redis minio
```

MySQL 首次创建数据卷时只创建空数据库，不再通过 Docker entrypoint 导入 `sql/schema_mysql8.sql` 或 `sql/seed_demo_data.sql`。

本地表结构和演示数据统一由后端 Flyway 执行：

1. `backend/src/main/resources/db/migration/V1__init_schema.sql`
2. `backend/src/main/resources/db/migration/V2__seed_demo_data.sql`
3. `backend/src/main/resources/db/migration/V3+` 后续增量迁移

默认数据库和账号：

| 项目 | 值 |
|---|---|
| 数据库 | `property_saas` |
| root 密码 | `root123456` |
| 应用账号 | `property` |
| 应用密码 | `property` |
| MySQL 宿主机端口 | `3307`，可通过 `MYSQL_PORT` 覆盖 |
| Redis 宿主机端口 | `6380`，可通过 `REDIS_PORT` 覆盖 |
| MinIO API 端口 | `19000`，可通过 `MINIO_API_PORT` 覆盖 |
| MinIO 控制台端口 | `19001`，可通过 `MINIO_CONSOLE_PORT` 覆盖 |

本地演示默认值可从 `.env.example` 复制：

```bash
cp .env.example .env
docker compose up -d mysql redis minio
```

`sql/schema_mysql8.sql`、`sql/seed_demo_data.sql` 和 `sql/seed_data.sql` 保留为历史交付/审阅兼容文件；本地开发运行时以 Flyway 迁移目录为准，避免初始化 SQL 与增量迁移双维护。

## 2.1 生产迁移目录

生产环境使用独立迁移目录：

```text
backend/src/main/resources/db/migration-prod
```

该目录包含建表基线、登录日志表和权限菜单迁移，不包含演示租户、演示账号、演示房屋、演示账单和演示支付配置。

生产后端启动示例：

```bash
SPRING_PROFILES_ACTIVE=prod \
DB_URL='jdbc:mysql://mysql:3306/property_saas?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai&useSSL=true' \
DB_USERNAME='property_prod' \
DB_PASSWORD='<from-secret-manager>' \
REDIS_HOST='redis' \
REDIS_PORT='6379' \
REDIS_PASSWORD='<from-secret-manager>' \
PROPERTY_SAAS_JWT_SECRET='<strong-secret>' \
java -jar property-saas-backend.jar
```

生产首个平台管理员可参考 `sql/seed_prod_minimal.sql`，替换占位符后执行。该模板只创建套餐、平台管理员和平台管理员角色，不创建演示租户。

后续数据库变更必须新增迁移文件，例如：

```text
V3__add_login_log.sql
V4__alter_sys_user_add_lock_fields.sql
```

不得直接修改已经发布使用的历史迁移文件。若还未进入真实开发环境，可同步修订 `sql/schema_mysql8.sql` 和迁移基线，但必须在任务交付中说明。

## 3. Docker 初始化与 Flyway 的关系

Docker 只负责启动本地基础设施。后端 Flyway 负责创建表结构、写入演示种子数据并执行后续增量迁移。

`application.yml` 中保留：

```yaml
spring.flyway.baseline-on-migrate: true
spring.flyway.baseline-version: 2
```

含义：兼容历史上已经通过初始化 SQL 建好的本地库；新的空库会由 Flyway 从 V1 开始完整执行到最新版本。

生产 profile 不使用上述本地基线设置；生产使用 `classpath:db/migration-prod`，并要求在空库上由 Flyway 从 V1 开始执行。

云端部署模板 `deploy/cloud/docker-compose.property-saas.yml` 也按生产 Flyway 口径处理，不再挂载 `sql/schema_mysql8.sql` 或 `sql/seed_demo_data.sql`。如云服务器已有旧演示库，必须先执行 `docs/deploy/云端数据库迁移切换方案.md`，确认备份和 `flyway_schema_history` 状态后再切换。

## 4. 重置本地数据库

本地需要重建全部表和种子数据时，可删除 MySQL 数据卷后重启：

```bash
docker compose down -v
docker compose up -d mysql redis minio
```

然后启动后端，Flyway 会自动执行完整迁移：

```bash
cd backend
mvn spring-boot:run
```

执行前必须确认本地数据可以丢弃。生产环境禁止用该方式重置。

## 5. 迁移约束

1. 所有租户业务表必须包含 `tenant_id`。
2. 项目级业务表必须包含 `project_id`。
3. 金额字段必须使用 `DECIMAL`，Java 侧使用 `BigDecimal`。
4. 账单、订单、支付流水、退款单、退款流水不得混表。
5. 新增状态枚举必须同步 `docs/business/06-业务状态机.md` 和 `openapi/openapi.yaml`。
6. 新增接口字段必须同步 `openapi/openapi.yaml`。
7. 高风险表变更必须说明数据迁移、回滚方式和接口兼容影响。
