# 00-开发前冻结事项

> 本文件为 Codex 正式开发前必须读取的冻结规则。若本文件与其他文档存在冲突，以本文件、根目录 `AGENTS.md`、`openapi/openapi.yaml`、`sql/schema_mysql8.sql` 的优先级为准。未经项目负责人明确确认，Codex 不得擅自修改本文件中的冻结项。

## 1. 技术版本冻结

| 类型 | 冻结决定 | 说明 |
|---|---|---|
| 后端语言 | Java | 不允许改为 PHP、Python、Node.js、Go 等其他语言。 |
| JDK | JDK 17 | Spring Boot 3.x 稳定组合，避免过度追新。 |
| 后端框架 | Spring Boot 3.2.x | 若依赖兼容需要可升至 3.3.x，但不得降至 Spring Boot 2.x。 |
| 权限认证 | Spring Security + JWT | 后台管理端使用 JWT；微信端使用微信登录换取系统 token。 |
| ORM | MyBatis-Plus | 不得擅自切换 JPA/Hibernate。 |
| 数据库 | MySQL 8.0.x | 字符集统一 `utf8mb4`；金额统一 `DECIMAL`。 |
| 缓存 | Redis 7.x | 所有业务 key 必须包含租户维度。 |
| 文件存储 | MinIO / S3 兼容对象存储 | 文件路径必须包含租户、项目、模块。 |
| 后端构建 | Maven 3.9.x | 不使用 Gradle，除非项目负责人明确确认。 |
| 管理端 | Vue 3 + TypeScript + Vite + Element Plus | 平台运营端与物业管理端使用同一个 PC 前端项目。 |
| Node.js | Node.js 20 LTS | 前端包管理器建议 pnpm，若工程模板固定 npm 需说明。 |
| 微信端 | uni-app + Vue3，目标为微信小程序 | 注意：用户口头提到的 `un-app` 按 `uni-app` 理解。 |
| API 文档 | OpenAPI 3.0 + Knife4j/Swagger | 实际接口必须与契约保持一致。 |
| 部署 | Docker Compose 优先 | 可预留 Kubernetes，但第一版以 Compose 可运行交付为准。 |

## 2. 前端端口与项目边界冻结

本系统第一版包含三个使用入口，但只建设两个前端工程：

| 入口 | 工程 | 使用对象 | 说明 |
|---|---|---|---|
| SaaS 平台运营端 | `admin-web` | 平台方管理员、运营、客服、财务 | 管理租户、套餐、租户状态、平台报表。 |
| 物业公司管理端 | `admin-web` | 物业公司总部、小区项目经理、财务、客服、工程、保安 | 与平台运营端共用一个 PC 管理端项目，通过角色、菜单、数据权限区分。 |
| 微信小程序端 | `mobile-uniapp` | 业主、住户、租户、家属、访客 | 使用 uni-app 开发，目标发布为微信小程序。 |

### 强制要求

1. 平台运营端与物业管理端不得拆成两个完全独立 PC 项目，除非项目负责人明确确认。
2. PC 管理端根据登录用户类型、角色、租户身份、项目权限动态生成菜单。
3. 后台员工账号使用 `sys_user`；微信业主/住户账号使用 `member_user`，不得混用。
4. 微信端只做业主/住户/租户侧功能，不承载物业员工后台管理能力。
5. `member_user` 冻结为租户内会员档案模型：同一微信用户跨多个租户时，按 `tenant_id + openid` 分别建立会员档案；跨租户统一身份或用户画像只能通过 `unionid` 等字段另行聚合，不得破坏租户数据隔离。

## 3. SaaS 多租户模式冻结

| 项目 | 冻结决定 |
|---|---|
| 多租户模式 | 共享库共享表 |
| 隔离字段 | `tenant_id` |
| 小区/项目字段 | `project_id` |
| 平台全局表 | 可不带 `tenant_id`，但必须在数据库设计说明中标注 |
| 租户业务表 | 必须带 `tenant_id` |
| 小区相关业务表 | 必须同时带 `tenant_id` 与 `project_id` |
| 租户来源 | 后端从登录态、token、租户上下文、支付订单反查，不信任前端直接传入 |
| 跨租户访问 | 默认禁止；平台端跨租户统计必须有专门接口、专门权限和审计日志 |

### 强制要求

1. Codex 不得改为“一租户一库”或“一租户一 schema”。
2. 所有查询、详情、更新、删除、导出、统计、定时任务、消息推送、文件下载、缓存、支付回调均必须考虑租户隔离。
3. 租户隔离不能只靠前端菜单隐藏，必须在后端数据层实现。
4. 所有跨租户风险必须加入测试用例。

## 4. 支付模式冻结

第一版账务和支付模型按以下原则设计：

| 项目 | 冻结决定 |
|---|---|
| 推荐主模式 | 微信支付服务商模式 |
| 兼容模式 | 租户自有普通微信商户号 |
| 不推荐模式 | 平台统一收款后人工结算给物业公司 |
| 支付归属 | 账单、订单、支付流水、退款流水均必须绑定 `tenant_id` |
| 商户配置 | 按租户维度配置，必要时支持项目级扩展 |
| 回调处理 | 必须验签、幂等、可追溯、可根据订单反查租户 |

已确认第一版支付口径：优先按微信支付服务商/子商户模式设计，同时兼容租户自有普通微信商户号；不得默认采用平台统一收款。未取得真实商户资料前，可在 dev profile 提供模拟支付，但数据库、接口、回调和退款流程必须按真实商户模式预留。

### 数据模型必须预留

至少预留以下能力，字段名可结合现有数据库脚本，但语义不得缺失：

```text
tenant_pay_config        租户支付配置
pay_channel              支付渠道，如 WECHAT
merchant_mode            商户模式：SERVICE_PROVIDER / NORMAL_MCH
mch_id                   普通商户号或服务商商户号
sub_mch_id               服务商模式下的子商户号
app_id                   小程序 appid
api_v3_key               微信支付 API v3 密钥密文或引用
cert_serial_no           商户证书序列号
settlement_mode          结算模式
status                   启用状态
```

### 强制要求

1. 账单是应收，支付订单是支付发起记录，支付流水是实收结果，退款流水是退款结果，四者不得混表。
2. 微信支付回调必须通过订单号或支付流水号反查租户，不得直接信任回调参数中的租户信息。
3. 不得把平台统一收款作为默认实现。
4. 未接入真实微信支付前，可以提供沙箱/模拟支付实现，但必须明确标注为 dev profile，不得污染生产逻辑。
5. `tenant_pay_config` 必须支持 `SERVICE_PROVIDER` 和 `NORMAL_MCH` 两类商户模式；支付下单、退款、回调验签、对账均必须按当前订单所属租户的支付配置执行。

## 5. 第三方设备接口策略冻结

门禁、梯控、停车、监控等设备系统第一版只做**适配层和接口预留**，不深度绑定具体厂商。

| 设备类型 | 第一版范围 | 禁止事项 |
|---|---|---|
| 门禁 | 设备档案、接口配置、权限下发记录、回调记录、访客通行记录 | 不写死某一家门禁厂商 SDK。 |
| 梯控 | 设备档案、接口配置、权限同步预留 | 不承诺完成所有梯控协议。 |
| 停车 | 车辆、车位、月租、第三方同步记录、入出场记录预留 | 不直接开发完整停车场硬件控制系统。 |
| 监控 | 设备档案、播放地址/平台地址预留 | 不直接开发视频监控平台。 |

### 强制要求

1. 设备相关模块必须抽象为 `DeviceAdapter` 或等价适配层设计。
2. 厂商接口参数必须配置化，不得硬编码。
3. 所有第三方调用必须记录请求、响应、状态、失败原因和重试次数。
4. 没有厂商接口文档和测试账号时，只允许完成适配层、配置表、模拟实现和回调记录表。

## 6. 开发方式冻结

| 项目 | 冻结决定 |
|---|---|
| 交付方式 | 一次性整体开发、整体联调、整体验收 |
| Codex 执行方式 | 按 TASK 文件逐个执行 |
| 首次任务 | 只读预审，不允许写代码 |
| 每次任务数量 | 一次只允许执行一个 TASK |
| 任务完成后 | 必须输出改动文件、实现内容、测试结果、未完成事项 |
| 进入下一任务 | 必须等待人工确认 |

### 强制要求

Codex 不得一次性执行 `TASK-001` 至 `TASK-022`。开发必须先完成基础工程、数据库、认证、多租户、权限，再进入业务模块。

## 7. 首次交给 Codex 的执行要求

第一次把本工程包交给 Codex 时，不要要求它开发系统。必须先让它执行：

```text
prompts/Codex首次只读预审提示词.md
```

目标是先发现工程包内部的字段、接口、权限、状态、任务顺序、测试用例之间的矛盾。只有预审问题处理完后，才允许执行 `TASK-001`。

## 8. 冲突优先级

若多个文件之间存在冲突，优先级如下：

```text
1. AGENTS.md
2. docs/00-开发前冻结事项.md
3. sql/schema_mysql8.sql
4. openapi/openapi.yaml
5. docs/business/05-权限矩阵.md
6. docs/business/06-业务状态机.md
7. tasks/TASK-xxx

```

Word 原始技术文档只作为背景参考，不作为 Codex 开发时的最高约束。
