# 03-多租户设计

## 1. 租户边界

租户是数据隔离、权限隔离、配置隔离、套餐控制和结算管理的根边界。一个租户通常对应一家物业公司或一个独立运营主体。

## 2. 推荐隔离模式

默认采用共享库共享表 + `tenant_id` 字段隔离。

| 模式 | 适用 | 本项目建议 |
|---|---|---|
| 共享库共享表 | 中小物业公司 SaaS 规模化 | 默认采用 |
| 共享库独立 Schema | 中大型租户 | 预留扩展 |
| 独立数据库 | 集团/国企强隔离客户 | 后续高级套餐 |
| 私有化部署 | 政企本地化客户 | 单独项目 |

## 3. 租户识别

| 入口 | 租户识别方式 |
|---|---|
| 平台运营端 | 平台用户不绑定单一租户，通过平台权限操作租户配置 |
| PC 租户管理端 | 后台账号绑定 tenant_id，Token 中携带租户上下文 |
| 移动管理端 | 员工账号绑定 tenant_id，并按 project_id 做数据范围 |
| 微信小程序 | 用户选择当前房屋后确定 tenant_id、project_id、house_id |
| 支付回调 | 根据 pay_order_no / 商户号 / appid 反查 tenant_id |
| 第三方回调 | 根据设备映射、接口配置、签名、回调路径反查 tenant_id |

## 4. 数据隔离要求

1. 业务表必须有 `tenant_id`。
2. 项目级数据必须有 `project_id`。
3. Mapper 层必须通过拦截器自动追加租户条件，复杂 SQL 必须显式写入租户条件。
4. 导出、统计、定时任务和异步任务不能绕过租户过滤。
5. 管理端不能通过 URL 参数切换租户。
6. 平台运维查看租户数据必须授权留痕。

## 5. 缓存隔离

Redis key 规范：

```text
tenant:{tenantId}:user:{userId}:menus
tenant:{tenantId}:project:{projectId}:dict:{dictType}
tenant:{tenantId}:bill:generate:{batchNo}
tenant:{tenantId}:payment:order:{orderNo}
```

禁止使用：

```text
bill:generate:{batchNo}
user:{userId}:menus
```

## 6. 文件隔离

对象存储路径：

```text
/tenant/{tenant_id}/project/{project_id}/module/{module}/yyyy/mm/{file_id}.{ext}
```

敏感文件下载必须经过后端鉴权，不能直接返回永久公开 URL。

## 7. 租户生命周期

| 状态 | 说明 | 系统行为 |
|---|---|---|
| PENDING | 待开通 | 不允许租户端登录 |
| TRIAL | 试用中 | 可使用套餐内功能 |
| ACTIVE | 正式启用 | 正常使用 |
| ARREARS_LIMITED | 欠费限制 | 可查看，限制新增/发送/生成等高成本操作 |
| SUSPENDED | 暂停服务 | 限制登录和接口调用 |
| ARCHIVED | 已归档 | 只保留数据，不允许业务操作 |

## 8. 套餐与额度

套餐控制维度：

- 模块开关；
- 项目数量；
- 后台账号数量；
- 房屋数量；
- 微信用户数量；
- 文件容量；
- 短信条数；
- 第三方接口调用量；
- 服务到期日。

所有套餐限制必须在后端校验，不能只依赖菜单隐藏。

## 9. 后端实现基线

TASK-004 起，后端多租户隔离采用以下实现基线：

1. `JwtAuthenticationFilter` 从认证态恢复 `userId`、`tenantId`、`userType`，并从 `X-Project-Id` 读取当前项目上下文。
2. `TenantContext` 保存当前用户、租户、项目和用户类型，请求结束后必须清理。
3. MyBatis-Plus 使用 `TenantLineInnerInterceptor` 自动追加 `tenant_id` 条件。
4. 平台全局表和公共日志表通过 `property-saas.tenant.ignore-tables` 配置白名单。
5. 平台用户默认不走自动租户条件；平台端跨租户查询必须使用专门接口、专门权限和审计日志。
6. 租户用户访问租户业务表时必须存在 `tenant_id`，否则后端应拒绝执行。
7. Redis key 统一通过 `TenantRedisKeyBuilder` 生成，格式为 `tenant:{tenantId}:...`。
8. 租户业务文件对象路径统一通过 `TenantObjectKeyBuilder` 生成，格式为 `/tenant/{tenantId}/project/{projectId}/module/{module}/yyyy/mm/{fileId}.{ext}`。
