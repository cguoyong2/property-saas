# 04-数据库设计说明

## 1. 命名规范

| 类型 | 规则 | 示例 |
|---|---|---|
| 表名 | 小写下划线，按领域前缀 | `sys_tenant`、`base_house`、`fee_bill` |
| 主键 | `{业务名}_id` 或 `id`，统一 BIGINT | `tenant_id`、`house_id` |
| 租户字段 | `tenant_id` | 所有业务表必有 |
| 项目字段 | `project_id` | 项目/小区级业务必有 |
| 平台/租户混合公共表 | `tenant_id` 允许为空时必须说明业务含义 | 平台公共消息、平台审计日志 |
| 金额字段 | DECIMAL，不使用 FLOAT/DOUBLE | `amount DECIMAL(12,2)` |
| 状态字段 | VARCHAR(32) 或 TINYINT，必须有枚举说明 | `status` |
| 逻辑删除 | `deleted TINYINT DEFAULT 0` | 0正常，1删除 |

## 2. 公共字段

除关系表和流水表特殊说明外，业务表默认包含：

```sql
created_by BIGINT NULL,
created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_by BIGINT NULL,
updated_at DATETIME NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP,
deleted TINYINT NOT NULL DEFAULT 0
```

## 4. 多租户例外说明

以下表或字段允许 `tenant_id` 为空，但必须在业务层做专门分支处理：

| 表 | 规则 |
|---|---|
| `sys_tenant_package` | 平台全局套餐表，不属于任何租户。 |
| `sys_menu` | 平台全局菜单/按钮/API 权限定义表，不属于任何租户。 |
| `sys_dept`、`sys_user`、`sys_role`、`sys_user_role`、`sys_role_menu` | 平台用户或平台角色数据允许 `tenant_id` 为空；租户用户、租户角色必须带 `tenant_id`。 |
| `sys_dict_type`、`sys_dict_data` | 平台全局字典允许 `tenant_id` 为空；租户自定义字典必须带 `tenant_id`。 |
| `file_object` | 平台公共文件允许 `tenant_id` 为空；租户业务文件必须带 `tenant_id`，项目级文件必须带 `project_id`。 |
| `message_record`、`operation_log`、`interface_call_log` | 平台消息、平台操作、平台级接口调用允许 `tenant_id` 为空；租户业务记录必须带 `tenant_id`。 |
| `message_template` | 租户级消息模板表，真实短信/微信模板 ID 等外部字段待正式厂商资料齐备后扩展。 |

## 6. 重要设计原则

1. 账单是应收，订单是支付请求，支付流水是第三方成功交易，退款流水是资金退回记录，四者不能合并。
2. 房屋与人员是多对多关系，不能简单把业主手机号写在房屋表里。
3. 租户、项目、房屋、账单、工单、文件之间必须能追溯。
4. 第三方设备接口必须有映射表和调用日志，不能只保存设备编号。
5. 所有导入必须有导入批次和错误记录，便于回滚或定位。
6. 第一版 SQL 默认不建立物理外键，避免 SaaS 多租户迁移、批量导入和逻辑删除场景下的部署耦合；但后端 Service 必须校验关键关联存在性，并通过唯一索引、普通索引和验收测试保证一致性。
