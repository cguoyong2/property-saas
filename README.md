# 智慧物业管理系统 SaaS 多租户版 Codex 工程包

## 一、工程包用途

本工程包由《智慧物业管理系统 SaaS 多租户一次性开发版技术文档 V1.1》转换而来，目标是让 Codex 在代码仓库中按固定边界自主开发，减少以下跑偏问题：

- 自行更换技术栈；
- 业务表遗漏 `tenant_id`；
- 平台端、物业端、业主端权限混用；
- 账单、订单、支付流水混淆；
- 接口路径、字段、状态、枚举自由发挥；
- 前端页面先写死 mock 数据，后端接口无法对接；
- 第三方门禁、停车、梯控接口被写死为单一厂商。

## 使用前强制要求

在把本工程包交给 Codex 后，第一次不要让它开发系统，也不要执行 `TASK-001`。

请先将 zip 解压到目标仓库根目录，或明确要求 Codex 从 zip 内读取文件。若未解压，直接引用 `prompts/...` 相对路径时，Codex 可能无法在当前工作区找到该文件。

首次必须让 Codex 执行：

```text
prompts/Codex首次只读预审提示词.md
```

目的：先让 Codex 只读审查工程包内部是否存在字段、接口、权限、状态机、任务顺序、验收用例之间的矛盾。预审问题处理完后，才允许进入 `TASK-001`。

本版本新增并冻结了 5 类关键事项：

1. 技术版本冻结；
2. 支付模式冻结；
3. 平台端 / 物业端 / 小程序端边界冻结；
4. 设备接口只做适配层，不做深度厂商绑定；
5. Codex 首次任务只读预审，不直接开发。

详见：`docs/00-开发前冻结事项.md`。

## 二、已冻结技术栈

- 后端：Java 17 + Spring Boot 3.2.x + Spring Security + JWT + MyBatis-Plus + Maven 3.9.x
- 数据库：MySQL 8.0.x，统一 utf8mb4，金额使用 DECIMAL(12,2)，主键使用 BIGINT
- 缓存：Redis 7.x，所有业务 key 必须带 tenant:{tenantId}: 前缀
- 文件：MinIO / S3 兼容对象存储，路径必须包含 tenant_id/project_id/module
- API 文档：OpenAPI 3.0 + Knife4j/Swagger UI
- 管理端：Vue 3 + TypeScript + Vite + Element Plus；平台运营端与物业管理端共用一个 PC 工程
- 微信端：uni-app + Vue3，目标平台为微信小程序
- 部署：Docker Compose 优先，预留 Kubernetes 扩展
- 日志：统一操作日志、接口日志、支付日志、异常日志

## 四、Codex 使用顺序

1. 第一次必须让 Codex 执行 `prompts/Codex首次只读预审提示词.md`，只读审查，不写代码。
2. 预审通过或问题修订后，再让 Codex 读取 `AGENTS.md`、`README.md`、`docs/00-开发前冻结事项.md`、`docs/00-CODEX使用说明.md`。
3. 再让 Codex 读取 `sql/schema_mysql8.sql`、`openapi/openapi.yaml`、`tasks/开发任务总清单.md`。
4. 每次只分配一个 `TASK-xxx` 文件给 Codex。
5. 每个任务完成后要求它运行自测，并对照 `tests/验收测试用例.md` 输出通过情况。
6. 不要一次性要求 Codex “开发完整系统”，应按任务顺序逐步完成。

## 五、关键红线

- 所有租户业务数据必须隔离。
- 所有金额必须可追溯到原始账单、订单、支付流水、退款流水。
- 所有高风险操作必须有权限、审批/确认、原因、日志。
- 所有接口必须与 `openapi/openapi.yaml` 保持一致。
