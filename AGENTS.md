# AGENTS.md

## 1. 项目定位

本仓库用于开发「智慧物业管理系统 SaaS 多租户版」。系统面向多家物业公司租户统一提供物业经营管理能力，包括平台运营端、物业租户管理端、移动管理端、微信小程序业主端。

本项目不是单物业公司私有系统，必须从第一行代码开始按 SaaS 多租户模型实现。

## 2. 技术栈固定

- 后端：Java 17 + Spring Boot 3.x + Spring Security + JWT + MyBatis-Plus + Maven
- 数据库：MySQL 8.x，统一 utf8mb4，金额使用 DECIMAL(12,2)，主键使用 BIGINT
- 缓存：Redis 7.x，所有业务 key 必须带 tenant:{tenantId}: 前缀
- 文件：MinIO / S3 兼容对象存储，路径必须包含 tenant_id/project_id/module
- API 文档：OpenAPI 3.0 + Knife4j/Swagger UI
- 管理端：Vue 3 + TypeScript + Element Plus
- 微信端：uni-app，目标平台为微信小程序
- 部署：Docker Compose 优先，预留 Kubernetes 扩展
- 日志：统一操作日志、接口日志、支付日志、异常日志

除非项目负责人明确批准，Codex 不得擅自更换后端语言、前端框架、数据库、ORM、权限框架、支付模式和目录结构。



## 2.1 开发前冻结事项

Codex 在执行任何开发任务前，必须读取并遵守 `docs/00-开发前冻结事项.md`。

以下事项已冻结，未经项目负责人明确确认不得变更：

1. 后端固定为 Java 17 + Spring Boot 3.2.x + Spring Security + JWT + MyBatis-Plus + MySQL 8.0 + Redis 7。
2. 管理端固定为 Vue 3 + TypeScript + Vite + Element Plus。
3. 微信端固定为 uni-app + Vue3，目标平台为微信小程序。
4. 平台运营端与物业管理端共用 `admin-web` 一个 PC 管理端工程，通过角色、菜单和数据权限区分。
5. SaaS 多租户固定采用共享库共享表 + `tenant_id` 隔离，不得改为一租户一库。
6. 支付模型按微信支付服务商模式设计，同时兼容租户自有普通商户号；不得默认采用平台统一收款。
7. 门禁、梯控、停车、监控第一版只做接口适配层和数据同步预留，不得写死具体厂商 SDK。
8. 一次只允许执行一个 `TASK-xxx`；完成后必须等待人工确认再进入下一任务。

## 2.2 首次执行规则

第一次使用本工程包时，Codex 必须先执行 `prompts/Codex首次只读预审提示词.md`，只输出《工程包预审问题清单》。首次任务禁止写代码、禁止修改文件、禁止初始化项目、禁止执行 `TASK-001`。

## 3. 强制多租户规则

1. 除平台全局表外，所有业务表必须包含 `tenant_id`。
2. 涉及具体小区/项目的数据必须包含 `project_id`。
3. 后端必须通过登录态/租户上下文自动确定 `tenant_id`，不得盲信前端提交的 `tenant_id`。
4. 查询、详情、更新、删除、导出、统计、定时任务、消息推送、文件下载、缓存读取、支付回调均必须做租户隔离。
5. 平台运营端可管理租户状态和配置，但不得绕过审计直接修改租户账务数据。
6. Redis key 必须使用 `tenant:{tenantId}:` 前缀。
7. 对象存储路径必须使用 `/tenant/{tenantId}/project/{projectId}/module/{module}/yyyy/mm/` 结构。
8. 支付回调必须能根据订单号或商户配置反查租户，不得从请求参数中直接信任租户信息。

## 4. 用户体系硬边界

- `sys_user`：平台运营人员、物业公司员工、项目员工等后台账号。
- `member_user`：微信端业主、住户、租户、家属、访客等用户。
- 后台账号和微信会员账号不得混用。
- 平台级用户、租户级用户、项目级用户必须通过角色权限和数据权限区分。

## 5. 开发顺序

Codex 必须按以下顺序推进，禁止跳过基础设计直接开发业务页面：

1. 读取 `README.md`、本文件、`docs/00-CODEX使用说明.md`。
2. 读取 `sql/schema_mysql8.sql` 和 `docs/architecture/03-数据库设计说明.md`。
3. 读取 `openapi/openapi.yaml`。
4. 先实现基础工程、认证授权、多租户上下文、租户管理、组织用户角色权限。
5. 再实现项目、小区、楼栋、房屋、业主、车辆等基础档案。
6. 再实现收费账务、支付、工单、公告、巡检、租赁合同、报表。
7. 最后实现第三方接口适配、性能优化、联调和验收测试。

## 6. 禁止事项

- 不允许接口返回 mock 数据后标记完成。
- 不允许业务表缺少 `tenant_id`。
- 不允许使用浮点数保存金额。
- 不允许把账单、订单、支付流水、退款流水混在一张表里。
- 不允许无权限校验地开放导出、退款、作废、账单修改等高风险接口。
- 不允许删除历史账务、合同、房屋绑定关系，只能作废、终止、失效或逻辑删除。
- 不允许为了快速通过测试而绕过租户隔离拦截器。
- 不允许硬编码门禁、梯控、停车某一家厂商接口，应走适配层。

## 7. 完成标准

每个模块完成必须同时满足：

- 数据库表/字段/索引已实现或确认不需要新增。
- 后端实体、Mapper、Service、Controller 已完成。
- 参数校验、权限校验、租户隔离、操作日志已完成。
- OpenAPI 文档与实际接口一致。
- 管理端/小程序端页面可调用真实接口。
- 至少包含正常、异常、越权、跨租户四类测试。
- `tests/验收测试用例.md` 中对应用例通过。

## 8. 修改既有设计的规则

如果 Codex 发现设计矛盾，必须先在输出中列出：

1. 冲突位置；
2. 影响范围；
3. 建议修改；
4. 是否涉及数据库迁移；
5. 是否影响接口兼容。

未经明确要求，不得直接修改已冻结的枚举、状态码、字段名、接口路径和租户隔离规则。
